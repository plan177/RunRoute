# RunRoute — Правила разработки

## Перед изменениями

- Прочитать `README.md` и затронутые файлы
- Понять архитектуру и текущее поведение

## Ветки и PR

- Одна задача — одна ветка и один PR
- Начинать ветку от актуального `master`
- Название ветки: `feat/`, `fix/`, `refactor/`, `docs/`, `perf/`

## Что нельзя делать

- Менять поведение вне текущей задачи
- Делать массовое форматирование
- Добавлять зависимости без явной необходимости
- Коммитить `.env`, токены и ключи
- Смешивать рефакторинг и новую функциональность

## Что нельзя менять

- `bot.py`
- Файлы `mini-app/` (если задача не связана с ними)
- Зависимости (`requirements.txt`)
- Конфигурацию деплоя (`railway.json`)
- CI/CD

## При bugfix

1. Описать причину бага
2. Описать как баг воспроизводится
3. Исправить
4. Перечислить проверки

## При рефакторинге

1. Подтвердить, что поведение не изменено
2. Перечислить затронутые функции
3. Проверить сценарии из README.md

## После изменений

- Перечислить проверки в описании PR
- Убедиться, что auto/manual/track работают
- Убедиться, что экспорт и шаринг работают

## Файлы проекта

| Файл | Назначение | Можно менять |
|------|-----------|--------------|
| `bot.py` | Telegram-бот | Нет |
| `mini-app/app.js` | Клиентская логика | Да (если задача требует) |
| `mini-app/index.html` | Разметка | Да (если задача требует) |
| `mini-app/style.css` | Стили | Да (если задача требует) |
| `backend/` | API сервер | Нет (не используется) |
| `requirements.txt` | Зависимости | Нет |
| `.env.example` | Пример конфигурации | Да |
| `.gitignore` | Правила игнорирования | Да |
